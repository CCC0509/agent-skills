# v0.5 Closeout Self-Report and Memory Routing Design

**Status:** Design spec for review.

**Goal:** Define a closeout self-report contract that tells agents how to
summarize completed work and route durable facts into existing repo-owned
memory without creating new memory types, templates, vault rules, or runtime
mechanisms.

## Problem

v0.3.0 established the repo-memory foundation: canonical memory is repo-owned,
`docs/agent-memory-index.md` names the repo's durable memory surfaces, and
status, lesson, audit, index, and mechanism evidence have distinct lifecycles.
Later v0.4 releases hardened review, relay, approval, and handoff behavior.

The remaining gap is closeout-time digestion. Agents can finish a task, pass
tests, and emit a relay block, yet still leave future sessions with either too
much stale chat context or too little durable state. The repo-memory protocol
says what memory types exist, but not how an agent should decide at closeout
which facts belong in status memory, lesson memory, audit memory, or nowhere.

This gap is related to several ROADMAP rows, but it must not collapse them into
one oversized release:

- The closeout half of `Preflight / closeout self-report contract` is the
  primary v0.5.0 scope. The preflight half stays as a narrowed follow-up unless
  the implementation proves the existing session-start memory-index rule fully
  covers it.
- `Named absence statuses for missing closeout evidence` depends on the
  self-report vocabulary and stays deferred so category names do not drift from
  Agent Trigger Kit.
- `Extraction-candidate closeout protocol` remains a separate follow-up. It is
  about triaging extraction candidates at closeout, not routing ordinary task
  facts into repo memory.

The practical user goal is also markdown-friendly project memory: lightweight
new-session seeds, active versus archived context, and notes that can be opened
from tools such as Obsidian. The portable rule must still keep repo-owned
markdown as the decision-grade source of truth.

## Design

Add a new subsection to `skills/agent-operating-manual/15-repo-memory.md`
called:

```text
Closeout self-report and memory routing
```

The subsection should be a routing procedure over the existing v0.3 memory
taxonomy. It must not introduce `phase archive`, `next-session seed`, or
`Obsidian vault` as new memory types. Those are patterns or accelerators:

- A next-session seed is a status-memory pattern.
- Closing or archiving old phase context is a lifecycle operation on status or
  audit memory, depending on the repo's `docs/agent-memory-index.md`.
- An Obsidian vault is a non-canonical accelerator or personal index unless the
  repo itself declares a path in `docs/agent-memory-index.md`.

The existing `Session protocol` list in `15-repo-memory.md` should remain the
compact session-start and session-closeout overview. The implementation should
cross-reference the new closeout subsection from that list instead of replacing
the list or duplicating a second full routing table there.

At closeout, the agent should first read the repo's memory index when present,
then classify each closing fact by destination:

| Destination | Use when | Do not use for |
|---|---|---|
| Status memory | Future agents need active state, next actions, blockers, current risks, or a compact next-session seed. | Permanent proof that work happened. |
| Lesson memory | A reusable pitfall, repeated correction, review finding pattern, or ATK closeout signal should teach future agents. | One-off task facts or ordinary completion notes. |
| Audit memory | The repo needs durable evidence of review, verification, deploy, smoke, production observation, or completion. | Active worklists or conversational context. |
| Index memory | Memory locations changed, or the first concrete lesson/status/audit path is chosen. | Routine task notes. |
| Mechanism evidence | ATK outcome stores, CI logs, MCP caches, command output, or scratch artifacts inform the closeout. | Canonical storage; summarize reusable facts into repo memory instead. |

The subsection should include a compact closeout self-report shape that an
agent can run mentally before final response or relay:

```text
Memory closeout:
- Status memory update: <none | path + one-line reason>
- Lesson memory update: <none | path + lesson class>
- Audit memory update: <none | path + evidence type>
- Index memory update: <none | path + location change>
- Mechanism evidence consumed: <none | source labels only>
- Next-session seed: <none | status-memory path>
```

The `Next-session seed` line highlights a status-memory subset. When a seed is
written, repeating the same path in `Status memory update` and
`Next-session seed` is expected; agents should not invent a separate seed file
just to make those lines differ.

This shape is a self-report discipline, not a new relay block and not a
required final-output field for every task. For ordinary small tasks where no
repo memory changes are warranted, the agent may keep the self-report internal
and mention `Memory closeout: none` only when the closeout itself is the user's
request or when the absence is important to avoid ambiguity.

When repo memory files are updated at closeout, the diff or commit is the
self-evidencing artifact. The final response does not need to print the
self-report shape unless the user requested a memory closeout report or the
handoff is a relay.

When the task already emits the Agent Operating Manual relay block, any memory
facts needed by the next agent should appear inside the forwarded context or in
`Accepted residuals:` if they are non-blocking residuals. The memory
self-report does not override `Status:`, `User action:`, `Blockers:`, or
`Accepted residuals:`.

## Routing Rules

The implementation should add rules with these semantics:

1. Read `docs/agent-memory-index.md` when present before deciding paths.
2. Route facts to existing memory types only. Do not create new peer types.
3. Prefer updating active status memory for facts that a near-future session
   needs to act safely.
4. Prefer audit memory for durable proof: review, verification, deploy, smoke,
   production observation, release, or completion evidence.
5. Prefer lesson memory only when the fact is reusable across future work.
6. Update index memory only when memory locations change or the repo chooses a
   concrete path that was previously absent.
7. Treat mechanism evidence as input. Do not paste raw logs, outcome stores,
   or cache data into portable doctrine.
8. If a closeout says no memory update is needed, that must mean no future
   agent needs active state, no reusable lesson appeared, and durable evidence
   is either unnecessary or already recorded by repo convention.

## Obsidian-Compatible Boundary

Repo-owned memory should remain portable markdown that tools such as Obsidian
can open and connect. For this release, "Obsidian-compatible" means plain
CommonMark-style markdown:

- Use normal relative markdown links when linking repo files.
- Do not require `[[wikilinks]]`, Dataview queries, YAML frontmatter, tags, or
  Obsidian plugins in canonical repo memory.
- A personal or cross-project vault may mirror or index repo memory, but it is
  an accelerator. Decision-grade facts flow from repo memory to the vault, not
  from the vault back into canonical memory unless an agent explicitly writes
  the corresponding repo-owned file through normal repo governance.

This preserves the user's "external brain" workflow without making any
particular local vault required for other agents.

## Pre-Spec Review Disposition

This spec incorporates the pre-spec framing review provided during planning.

- F1, ROADMAP mandate mismatch: accepted. The release is framed as closeout
  self-report plus memory routing, absorbing the closeout half of the
  `Preflight / closeout self-report contract` candidate. The preflight half
  remains a narrowed follow-up unless implementation review confirms existing
  session-start memory-index guidance fully covers it. The
  `Extraction-candidate closeout protocol` row stays deferred and must not be
  treated as implemented by this release.
- F2, memory taxonomy drift: accepted. The design routes to existing v0.3 memory
  types only. `Phase archive` and `next-session seed` are patterns, not new
  memory types.
- F3, doctrine-only first increment: accepted. Starter templates, installer
  file creation, and vault conventions are deferred.
- F4, Obsidian-compatible boundary: accepted. Canonical files stay portable
  CommonMark; vaults are accelerators, not sources of truth.
- F5, duplication guardrails: accepted. The implementation should reference
  the existing lifecycle table instead of restating it as a second normative
  home.
- F6, archive link-rot risk: accepted. If a repo moves status or audit files as
  part of archiving, the same change must update `docs/agent-memory-index.md`
  or the repo should prefer marking closed in place.
- Review-format note: the pre-spec review copy available during spec drafting
  omitted the required `Review stance:` opening line. Treat that as a relay /
  copy-format miss, not as a blocker for the substance of the findings.

## Scope

In scope:

- Add the closeout self-report and memory-routing subsection to
  `skills/agent-operating-manual/15-repo-memory.md`.
- Add a short pointer from `skills/agent-operating-manual/README.md` and
  `skills/agent-operating-manual/SKILL.md` if needed so agents can discover
  the closeout rule from the memory section.
- Update `tests/install-smoke.sh` with stable imported-manual tokens for the
  new subsection and representative terms.
- Add a v0.5.0 landed entry to `ROADMAP.md`.
- Rewrite the `Preflight / closeout self-report contract` candidate to a
  preflight-only remainder after the landed entry exists, unless implementation
  review confirms the existing session-start memory-index guidance fully covers
  the preflight half.
- Keep the `Named absence statuses for missing closeout evidence` row deferred,
  with wording that it waits for this self-report baseline plus ATK taxonomy
  alignment.
- Keep the `Extraction-candidate closeout protocol` row deferred as a distinct
  follow-up, correcting the prior `schedule it for v0.5.0` wording.
- Add or preserve follow-up rows for starter templates, Obsidian vault
  convention, brainstorming decision-assist, review deliverable handoff
  copy-field tightening, and branch / worker lifecycle.
- Bump `.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json` to
  `0.5.0` during implementation. This follows the current v0.4 release-train
  pattern where implementation records the next release metadata while release
  tagging and publishing remain out of scope until explicit approval.

Out of scope:

- Creating starter memory templates.
- Editing adopting repos or generated imported skill copies directly.
- Changing `install.sh` to create new memory files beyond the existing
  `docs/agent-memory-index.md` baseline.
- Adding validator, session-check, closeout checker, hook, or outcome taxonomy
  mechanisms. Agent Trigger Kit owns those.
- Defining named absence categories.
- Defining extraction-candidate triage rules.
- Defining branch, worker, subagent, worktree, or local-branch lifecycle
  cleanup rules.
- Defining Obsidian vault layout, sync scripts, backlinks, Dataview, tags, or
  frontmatter.
- Making brainstorming decision-assist or pre-spec advisor review mandatory.
- Tightening review deliverable handoff copy-block requirements.
- Release tagging or publishing `v0.5.0`.

## Deferred Follow-Ups

These are intentionally not part of v0.5.0:

| Follow-up | Why deferred | Likely home |
|---|---|---|
| Brainstorming decision-assist protocol | Helps when the user or agent cannot choose during brainstorming, but it is a design-time advisory workflow rather than closeout memory routing. | agent-skills doctrine |
| Starter memory templates | Useful after the routing protocol stabilizes, but templates would create stronger file-shape opinions than this release needs. | agent-skills or adopting repos |
| Obsidian vault convention | Cross-project vault layout and backlink conventions are personal or organization-specific; canonical repo memory remains portable markdown. | adopting repo or separate doctrine candidate |
| Extraction-candidate closeout protocol | Its subject is candidate triage, not ordinary task memory routing. | agent-skills doctrine |
| Named absence statuses | Needs the closeout self-report baseline and Agent Trigger Kit taxonomy alignment. | agent-skills / Agent Trigger Kit |
| Preflight self-report contract | The v0.5.0 scope handles closeout memory routing; any remaining preflight self-report semantics should be designed after checking whether the existing session-start memory-index rule already covers them. | agent-skills doctrine |
| Review deliverable handoff copy-field tightening | Completed review reports can be mistaken for handoffs when they omit a paste-ready relay/copy area; this needs a separate control-contract pass over multi-angle-review and Agent Operating Manual §3.1. | agent-skills doctrine |
| Branch / worker lifecycle hygiene | It concerns worker spawn, wait, consume, close, post-merge push state, and cleanup, not memory closeout. | agent-skills doctrine |

## Review Notes

- The value of this release is the closeout decision procedure. It should help
  agents write less chat history and more durable, correctly routed memory.
- The release should not duplicate the v0.3 lifecycle table. It should point to
  that table and add the missing closeout-time routing step.
- The closeout self-report is a thinking/checking shape. It should not become a
  noisy required block in every final response.
- The Obsidian-compatible rule is intentionally conservative: ordinary markdown
  links and repo-owned files give the user an external-brain workflow without
  locking future agents to one local app.
