# D. 判斷力外化（Judgment Rubrics）

> 這一節把「強模型憑直覺做、弱模型抓不準」的判斷，寫成**可勾稽的規則 + 正反例**。
> 用法：把每條當 checklist。碰到對應情境時，**先對照這裡，再動作**。
> 全部是**跨 harness 通則**，可直接沿用。

---

## §1 何時升級模型

**規則**：升級的訊號是「**換了做法還再犯同一類錯**」（＝推理/能力缺口），不是「錯了一次」。一次性小失誤讓它自己修；反覆撞牆才升級。

- ✅ **正例**：sonnet subagent 交的修法，跑測試連 fail 兩次、**兩次都卡在同一類錯**（例如同一個型別推導過不了），換了兩種寫法還是犯同一類 → 推理/能力缺口，帶上兩次完整軌跡升 opus。（注意：若是**每次冒出不同的新錯**，那是「方向錯」訊號，走 §4 換路，不是升級。）
- ❌ **反例**：sonnet 的修法第一次 fail，原因是 `import` 路徑打錯、一看就能自己更正 → 讓它修一次，**不要升級**（升級只是浪費成本）。

---

## §2 何時算真的「完成」（Done）

**全部勾到 yes 才算 done**：

- [ ] 驗收條件**逐條**被**新 context 的 agent** 或**機器**確認過（不是做事的人自己說）。
- [ ] 程式碼：測試通過 **且** 實際跑過受影響的流程（**不是**「compile 過 / 看起來會動」）。
- [ ] 沒有殘留 TODO / stub / 註解掉的半成品。
- [ ] 範圍剛好對齊需求——**沒多做**（無關改動）也**沒少做**。
- [ ] 若動到既有檔：沒踩到別人的 dirty change；該同步的 docs 有同步。
- [ ] 若開了 worktree / branch：**已收尾**——merge 後移除**已 merge 的 worktree**、清理已 merge 的本機 branch；跑 full-repo test / check 前先移除已 merge worktree（部分 test runner 會掃進殘留 worktree、誤跑 live 測試——各 repo 具體風險見其 conventions addenda）。

- ✅ **正例**：「加一個欄位 X」→ 欄位加了、測試綠、fresh agent read-back 引出那幾行確認欄位在、`git diff` 無無關改動 → done。
- ❌ **反例**：宣稱 done 只因「code 會 compile」，但那條實際流程**從沒跑過** → **不是 done**。這是最傷的假性完成（見 `00-diagnosis.md` Leak 3）。

## §3 何時停下問使用者

**規則**：問，當符合任一——
- (a) 決策**不可逆且模糊**：刪資料、部署、覆蓋使用者/其他 agent 的工作、動到真錢邊界。
- (b) 需求**真的欠指定**，且猜錯會浪費一大段工。
- (c) 兩個都合理的解讀，會導出**差很多**的結果。

**不要問**：存在合理預設、且改起來很便宜的小決定——**選一個、講一聲、往前走**。

- ✅ **正例**：「這三個檔哪個該刪，我不確定哪個是你要保留的」→ 問。「要不要真的下單 / 部署到正式環境」→ 問。
- ❌ **反例**：「log 要寫 `fetched` 還是 `retrieved`？」→ 自己選，別問。為這種事停下＝把你的判斷責任推回給人。

**問的方式**：給**建議選項 + 各自後果**，不要丟開放式「你想怎樣？」。（採用 repo 若有自訂的不可逆邊界——例如部署、下單、刪 branch、動風控閾值——**一律歸 (a)，先問**；細節見該 repo 的 conventions addenda。）

## §4 什麼訊號代表方向錯了、該換路（不是重試）

**出現任一就停**，回去找 root cause 或問人，**別再 patch**：
- 同一錯誤類別，**換了兩種做法還再犯**。
- 每補一個 patch **就冒出一個新破壞**（打地鼠）。
- 你在「**跟框架/工具對打**」——一直繞過它的預設行為才能勉強動。
- 驗收標準**一直在移動**（每次快到就發現又漏一塊）。

- ✅ **正例**：為了讓一個測試過，改了三版、**每版都弄壞另一個測試** → 設計錯了。停止 patch，改做 systematic-debugging 或重新界定需求。
- ❌ **反例**：一個 assertion fail、原因清楚（少一個 `await`）→ 就是修它。這**不是**方向錯，別小題大作換路。

> 區分關鍵：**「同一類錯反覆出現」= 方向或能力問題**——先判 approach：**目標清楚、approach 可行、只卡在一個難點 → 升級模型（§1）**；**approach 本身在跟現實/框架對打、或你不確定它到底能不能成 → 換路／重新界定／問人（本節）**。而**「不同的、原因明確的錯逐一被解掉」= 正常進展**（繼續）。

## §5 品質底線怎麼驗（沒有強模型判斷時）

按產物類型，用**機器 + 第二眼**取代「我覺得沒問題」：

| 產物 | 驗法 |
|---|---|
| 程式碼 | 測試 + 實跑 + reviewer agent 帶**具體 checklist**：沒 swallow error、輸入有驗證、風格與周邊一致、沒硬編 secret、沒殘留 debug log |
| 文件 / 規則 | fresh agent read-back + 一遍「**弱模型會不會誤讀這句？**」的對抗式檢查 |
| 資料 / 數字 | 用**另一條獨立路徑**重算或抽樣核對，不接受單一來源自證 |

- ✅ **正例**：改完 scraper，派 reviewer agent 帶上表 checklist 逐條回 yes/no + `file:line`，再實跑一次抓資料核對 → 有機器與第二眼把關。
- ❌ **反例**：自己讀一遍自己寫的 code 說「看起來沒問題」→ 這不是驗證，是自我安慰。

---

## §6 誠實條款：機器驗不了的東西怎麼辦

有些事**拆解 + 驗證 + 多樣本評審都補不了**——品味、模糊的產品取捨、「這是不是對的抽象」。遇到時**不要假裝有把握**，三選一：

1. **產生 2–3 個選項 + 各自 trade-off，交使用者選**（品味題的正解）。
2. **升到最強可用模型要第二意見**（提高機率，但仍非保證，要標明是意見不是定論）。
3. **明說「這需要人的判斷 / 我無法可靠驗證這一點」**——標註出來，然後往下走或停。

- ✅ **正例**：「這個模組該不該拆成兩層？」沒有任何測試能判對錯 → 給兩種設計 + 取捨，讓使用者定。
- ❌ **反例**：對一個純品味問題，自己選一個還宣稱「這是最佳做法」→ 假確定。這正是誠實條款要擋的。

> 一句總結：**執行品質**可以靠拆解 + 驗證 + 多樣本補到很高；**品味與模糊判斷**補不了——那時的正確動作是「升級 / 要第二意見 / 明說做不到」，不是硬撐一個假答案。
